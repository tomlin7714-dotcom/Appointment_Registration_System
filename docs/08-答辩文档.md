## 二、技术栈选型说明（核心问题）

### Q: 为什么没有用课设要求的 Vue3 + Unibest？

**回答要点：**

1. **时间因素**：项目功能复杂（预约、排班、叫号、问诊、AI导诊等），需要用最短时间实现完整业务流程。Vue3和Unibest需要额外学习成本，原生HTML/JS开发效率更高。

2. **技术验证**：项目的核心价值在后端业务逻辑（预约流程、号源管理、支付状态机等），前端框架的选择不影响功能完整性。先用原生实现保证功能，后续再考虑技术栈升级。

3. **微信小程序原生**：微信小程序本身就有完善的原生开发框架，对于课设级别的项目，原生框架完全够用，且调试更方便。

4. **分工说明**：管理端和医生端是纯功能型页面（表格、表单、弹窗），用原生JS足以实现良好交互，不需要引入重型框架。

### Q: 密码为什么是明文存储？

**回答要点：**
- 课设阶段为方便演示和调试，未做加密
- 生产环境应该使用 BCrypt 或 Spring Security 的 PasswordEncoder 加密
- 加密改造不复杂，只需要在注册时加密、登录时匹配即可

### Q: 系统安全性如何？

**回答要点：**
- 使用了 Spring Security 做基础防护
- API Key 通过 `application-secret.properties` + `.gitignore` 隔离，不提交到 Git
- 后续可加入 Token 鉴权、接口限流、SQL注入防护等

## 三、核心业务逻辑速记

### 3.1 预约流程

```
用户选时段 → 系统检查号源 → 创建预约(状态=0待支付)
  → 用户支付 → 状态=1待就诊 → 医生叫号 → 状态=2就诊中
  → 医生填写诊断保存 → 状态=3已完成
  → 用户取消 → 状态=4已取消，号源释放
```

**关键点：** 号源扣减在创建预约时完成，取消时自动释放。

### 3.2 问诊单流程

```
用户预约时填写(问诊类型/主诉/病史) → 保存到 consultation_form
  → 医生叫号后查看 → 医生填写诊断/治疗 → 保存并完成就诊
```

**关键点：** `consultation_form` 与 `appointment` 是 1:1 关系。

### 3.3 排班逻辑

```
自动排班：遍历未来14天 → 跳过休息日(每位医生2天/周)
  → 每天4个时段(上午2个+下午2个) → 每个时段创建号源20个
```

**关键点：** 休息日由医生ID计算，不同医生不同休息日。排班只能新增不能覆盖。

### 3.4 AI导诊

```
用户输入症状 → 如果有API Key → 调DeepSeek API
  → 解析JSON返回 → 匹配数据库科室 → 展示推荐+建议
  → 如果没有Key → 关键词匹配 → 返回匹配科室
```

**关键点：** API不可用时自动降级，双模式保证功能可用。

## 四、数据库核心关系

```
dept (1) ──→ (N) doctor ──→ (N) schedule ──→ (N) number_source
                                                    │
user ──→ appointment ←── number_source (N:1)
             │
             └── consultation_form (1:1)
```

- `appointment.status`：0=待支付 1=待就诊 2=就诊中 3=已完成 4=已取消
- `consultation_form.status`：0=待诊断 1=已完成

## 五、常见问题解答

### Q: 项目有哪些创新点？

1. **AI导诊功能** — 接入DeepSeek API，用户输入症状即可推荐科室
2. **完整的问诊流程** — 从患者填写症状到医生诊断，全流程数字化
3. **自动排班系统** — 启动时和每日凌晨自动生成14天排班
4. **多端协同** — 小程序+PC浏览器，覆盖不同使用场景

### Q: 开发中遇到的最大困难是什么？

**问诊单表字段不一致**：最开始的代码假设了错误的数据表字段名（如 `consultation_type` 实际是 `form_type`），导致整个问诊单模块的实体、Mapper、Controller、前端都要重写。解决方案是先通过后端API查询实际表结构，再逐层修正代码。

### Q: 系统可以怎么扩展？

1. **接入真实支付** — 微信支付/支付宝 SDK
2. **消息通知** — 预约成功/就诊提醒短信或模板消息
3. **在线问诊** — 视频/图文咨询功能
4. **数据统计** — 接入ECharts做可视化报表
5. **多医院支持** — 扩展为平台级应用

### Q: 如何保证预约不超卖？

号源扣减采用"先检查再更新"策略：
1. 创建预约前查询 `remain_num > 0`
2. 创建预约记录
3. 执行 `UPDATE number_source SET remain_num = remain_num - 1`
4. 事务保证原子性

### Q: 排班规则是什么？

- 自动生成未来14天的排班
- 每位医生每周休息2天（由ID计算，不同医生错开）
- 每天分4个时段：08:00-10:00、10:00-12:00、14:00-16:00、16:00-18:00
- 每个时段默认20个号源
- 周日全体休息
