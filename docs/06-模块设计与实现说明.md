# 医疗预约挂号系统 —— 模块设计与实现说明

> 本文档用于快速掌握项目各功能模块的设计思路、实现要点和设计原因。

---

## 一、系统架构设计

### 1.1 整体架构

```
┌─────────────────────────────────────────────────┐
│                  前端层                          │
│  ┌──────────────┐  ┌──────────┐  ┌───────────┐  │
│  │ 微信小程序端 │  │ 管理端   │  │ 医生端    │  │
│  │  (患者使用)  │  │ (HTML)   │  │ (HTML)    │  │
│  └──────┬───────┘  └────┬─────┘  └─────┬─────┘  │
└─────────┼───────────────┼──────────────┼─────────┘
          │               │              │
          └───────────────┼──────────────┘
                          │ HTTP/JSON
                   ┌──────┴──────┐
                   │  Controller │  ← 接收请求、返回响应
                   ├──────┬──────┤
                   │Service│     │  ← 业务逻辑（本项目直接在Controller层实现）
                   ├──────┴──────┤
                   │   Mapper   │  ← 数据库访问
                   ├────────────┤
                   │   MySQL    │  ← 数据存储
                   └────────────┘
```

**为什么这么设计？**
- **三层架构**：Controller(接口层) → Mapper(数据层) → DB，层次清晰
- **未单独抽取 Service 层**：项目为课设规模，业务逻辑不复杂，直接在 Controller 中处理减少文件数量
- **统一返回值 ResponseVo**：所有接口返回统一格式 `{code, msg, data}`，前端统一处理

### 1.2 技术选型理由

| 技术 | 为什么选它 |
|------|-----------|
| Spring Boot | 快速搭建、内嵌Tomcat、自动配置、课设主流框架 |
| MyBatis | SQL灵活，多表JOIN查询方便，XML映射清晰 |
| MySQL | 免费、稳定、使用广泛 |
| 原生HTML/JS | 非课设要求的Vue3，但时间紧迫下开发最快，且管理端/医生端是功能型页面，不需要复杂交互 |
| 微信小程序原生 | 非Unibest，但原生框架调试方便、文档完善 |

## 二、用户模块设计

### 2.1 功能概述

用户注册、登录、信息管理、头像上传。

### 2.2 表结构

```sql
user (id, phone, pwd, real_name, id_card, avatar, status, create_time, last_login_time)
```

### 2.3 核心逻辑

**注册：**
```
前端提交 → 校验手机号唯一性 → 插入用户记录 → 返回成功
```

**登录：**
```
前端提交 → 查手机号 → 校验密码 → 更新last_login_time → 返回token+用户信息
```

### 2.4 设计要点

| 要点 | 说明 | 为什么 |
|------|------|--------|
| 手机号为登录账号 | 唯一标识 | 手机号天然唯一，用户易记 |
| 密码明文存储 | 不做加密 | 课设阶段方便调试，生产应加密 |
| status字段 | 0=启用 1=禁用 | 管理员可控制用户登录权限 |
| 头像单独接口 | `/api/user/avatar/upload` | MultipartFile上传，返回URL保存到数据库 |

## 三、科室与医生模块设计

### 3.1 功能概述

科室增删改查（含区域、诊室号），医生增删改查（含性别、职称），医生自动排班。

### 3.2 表关系

```
dept (1) ──→ (N) doctor
```

### 3.3 核心逻辑

**科室管理：**
```
增删改查都是标准CRUD，管理端操作
area和roomNumber字段用于标识就诊位置
```

**医生管理：**
```
增删改查 + 性别字段
dept_id外键关联科室
支持按关键词、科室、状态筛选
```

**自动排班（核心设计）：**
```
系统启动时(ScheduleInitializer) + 每日凌晨(ScheduleAutoTask)
  → 遍历所有医生
  → 遍历未来14天
  → 跳过休息日（由医生ID计算，不同医生错开）
  → 每天4个时段（08:00-10:00, 10:00-12:00, 14:00-16:00, 16:00-18:00）
  → 每个时段创建排班 + 号源20个
```

### 3.4 设计要点

| 要点 | 说明 | 为什么 |
|------|------|--------|
| 区域+诊室号 | dept表附加字段 | 方便患者找到就诊位置 |
| 休息日由ID计算 | restDay = doctorId % 7 + 1 | 自动分配，不用手动设置 |
| 周日固定休息 | 硬编码 | 大部分医院周日门诊少 |
| 排班不覆盖 | 已有时段跳过 | 防止手动排班被自动覆盖 |

## 四、预约挂号模块设计（核心）

### 4.1 功能概述

患者选择科室→日期→医生→时段，填写问诊信息，提交预约，支付，取消。

### 4.2 表关系

```
number_source (N) ←── appointment ──→ user
                    ↓
              schedule
```

### 4.3 核心逻辑

**预约流程（状态机）：**

```
创建预约 → status=0(待支付)
   ↓ 用户支付
status=1(待就诊)
   ↓ 医生叫号
status=2(就诊中)
   ↓ 医生完成诊断
status=3(已完成)

取消 → status=4(已取消)
```

**号源扣减：**
```
1. 查询 remain_num > 0
2. INSERT appointment 记录
3. UPDATE number_source SET remain_num = remain_num - 1
```

### 4.4 关键代码路径

```
前端(booking.js) → POST /api/user/appointment/create
  → AppointmentController.createAppointment()
    → 检查号源 → 创建预约 → 扣减号源 → 返回appointmentId
  → 前端调用 POST /api/user/consultation/save (保存问诊单)
  → 前端调用 POST /api/user/appointment/pay (支付)
```

### 4.5 设计要点

| 要点 | 说明 | 为什么 |
|------|------|--------|
| 状态机设计 | 5个状态明确 | 清晰追踪预约生命周期 |
| 号源先查后扣 | 非事务但简单 | 课设阶段不会高并发 |
| 问诊单独立保存 | 先预约后填问诊 | 解耦预约和问诊流程 |
| 取消释放号源 | remain_num +1 | 保证号源不浪费 |

## 五、问诊单模块设计

### 5.1 功能概述

用户填写症状信息（问诊类型、主诉、现病史、既往史），医生填写诊断（检查、诊断、治疗方案），用户查看问诊历史。

### 5.2 表结构

```sql
consultation_form (
  id, consultation_no, appointment_id, user_id, doctor_id,
  form_type, chief_complaint, present_illness, past_history,
  examination, diagnosis, treatment, status, create_time, update_time
)
```

### 5.3 核心逻辑

**问诊单生命周期：**
```
预约时填写 → consultation_form.appointment_id = 预约ID
  (或预约后在问诊历史中补充)
  → 医生查看、填写诊断 → status=1(已完成)
  → 用户查看问诊历史
```

### 5.4 双角色操作

| 角色 | 操作 | 字段 |
|------|------|------|
| 患者(用户) | 填写 | form_type, chief_complaint, present_illness, past_history |
| 医生 | 填写 | examination, diagnosis, treatment |
| 系统 | 自动 | consultation_no, status |

### 5.5 设计要点

| 要点 | 说明 | 为什么 |
|------|------|--------|
| 1:1关联预约 | appointment_id唯一 | 一次预约对应一份问诊单 |
| 编号规则 | CF + 时间戳 | 唯一且可读 |
| 角色分离 | 患者填症状、医生填诊断 | 权限清晰，各司其职 |
| 状态独立 | consultation_form有自己status | 与预约状态解耦 |

## 六、诊间叫号模块设计

### 6.1 功能概述

医生叫号、语音播报、状态流转、完成就诊。

### 6.2 核心流程

```
医生点击"叫下一个"
  → 查询今日所有status=1(待就诊)的预约
  → 按预约时间排序，取第一个
  → 调 startCall 接口 → status=2(就诊中)
  → 语音播报
  → 医生查看问诊单 → 填写诊断 → 调 saveConsultation
  → status=3(已完成)
```

### 6.3 设计要点

| 要点 | 说明 |
|------|------|
| 不依赖排队序号 | 直接用预约时间排序，先到先看 |
| 语音播报 | 使用浏览器 SpeechSynthesis API |
| 重叫功能 | 重新播放语音，不改变状态 |
| 跳过功能 | 当前患者跳到队尾（仅前端调整） |

## 七、管理员管理模块设计

### 7.1 功能概述

系统概览、科室管理、医生管理、用户管理、问诊记录管理。

### 7.2 各子模块

**系统概览：**
```
聚合查询：
SELECT COUNT(*) FROM dept → deptCount
SELECT COUNT(*) FROM doctor → doctorCount
SELECT COUNT(*) FROM user → userCount
今日预约 = appointment JOIN schedule WHERE visit_date = today
```

**用户管理（查询+CRUD）：**
```
查询条件：手机号、姓名、注册时间范围
排序：按ID降序（最新注册在前）
分页：每页5条
```

**问诊记录管理：**
```
查询条件：医生姓名、创建时间范围
分页：每页9条
编辑弹窗：分区域展示(基础信息/患者信息/医生信息/问诊信息/诊断信息)
```

### 7.3 设计要点

| 要点 | 说明 | 为什么 |
|------|------|--------|
| 每页5/9条 | 用户5条，问诊9条 | 数据量不同，合理展示 |
| 分页在Java层实现 | 非SQL分页 | 数据量小，代码简单 |
| 编辑弹窗区域化 | 分区展示信息 | 信息量大，分区更清晰 |

## 八、AI导诊模块设计

### 8.1 功能概述

用户输入症状，AI推荐科室和就医建议。

### 8.2 双模式设计

```
模式一：DeepSeek API（需配置Key）
  用户输入 → 调DeepSeek Chat API → 解析JSON返回
  → 匹配数据库科室 → 展示推荐

模式二：关键词匹配（无Key时自动降级）
  用户输入 → 遍历关键词规则表
  → 匹配科室 → 展示推荐
```

### 8.3 设计要点

| 要点 | 说明 | 为什么 |
|------|------|--------|
| 双模式 | API + 关键词 | API不可用时保证功能可用 |
| 科室匹配优先长名称 | 数据库按名称长度降序匹配 | 避免"呼吸内科"被"内科"抢匹配 |
| JSON解析+容错 | 解析AI返回的JSON，失败走降级 | AI可能返回非标准格式 |
| 一键预约 | 推荐科室可直接点"立即预约" | 减少用户操作步骤 |

## 十、快速理解要点

### 10.1 核心表关系一句话

- **预约是核心**：连接用户、号源、问诊单
- **排班产生号源**：schedule → number_source
- **问诊单绑定预约**：1:1关系
- **科室包含医生**：dept → doctor

### 10.2 关键状态值

**预约状态 (appointment.status)：**
```
0 = 待支付 → 1 = 待就诊 → 2 = 就诊中 → 3 = 已完成
                                    ↘ 4 = 已取消
```

**问诊单状态 (consultation_form.status)：**
```
0 = 待诊断 → 1 = 已完成
```

### 10.3 常用API速记

| 操作 | 接口 |
|------|------|
| 预约 | POST /api/user/appointment/create |
| 支付 | POST /api/user/appointment/pay |
| 取消 | POST /api/user/appointment/cancel |
| 保存问诊 | POST /api/user/consultation/save |
| 医生保存诊断 | POST /api/doctor/consultation/save |
| 叫号 | POST /api/doctor/call/start |
| 自动排班 | POST /api/doctor/schedule/auto |
| 用户列表 | GET /api/admin/user/list |
| 问诊记录列表 | GET /api/admin/consultation/list |
