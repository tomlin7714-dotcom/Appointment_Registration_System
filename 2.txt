一、引言
1.1 编写目的
本说明书详细定义医疗预约挂号系统的功能、性能、接口、数据等各方面需求，为系统的设计、开发、测试、部署及维护提供统一的标准和依据，确保开发团队、测试人员及相关方对系统需求达成共识，保障系统开发工作有序、高效开展，最终实现满足医院管理员、医生和患者实际使用需求的医疗预约挂号系统。
1.2 项目背景
在传统医疗服务中，患者线下排队挂号存在耗时久、流程繁琐、号源信息不透明等问题，医院也面临着挂号窗口工作压力大、号源管理效率低、医生排班与挂号信息同步不及时、基础信息管理杂乱等痛点。随着互联网技术在医疗领域的普及，线上预约挂号成为提升医疗服务效率、改善患者就医体验的重要方式，同时医院也需要专业的管理角色实现对全院基础信息、用户及医生账号的统一管控。
本项目为《软件工程综合实践》课程设计项目，基于 Java SpringBoot、MyBatis、MySQL、Vue 等技术开发医疗预约挂号系统，实现系统管理员网页端的基础信息管理、账号管理、系统监控等核心功能，医生网页端的个人信息管理、排班设置、号源发布等核心功能，以及用户小程序端的登录注册、预约挂号、记录查看等核心功能，打通线上挂号全流程，实现医院管理、医生工作、患者就诊的数字化协同，满足课程设计中独立完成小型软件系统开发的要求。

二、总体描述
2.1 产品愿景
打造一款操作简便、功能完善、性能稳定的医疗预约挂号系统，解决传统线下挂号的痛点，实现患者线上便捷预约挂号、医生高效管理号源与排班、管理员统一管控医院基础信息，预约流程自动化无审核，提升医疗服务整体效率和患者就医体验，同时满足《软件工程综合实践》课程设计的开发要求，完成小型软件系统的全流程开发。
2.2 功能概述
本系统分为系统管理员网页端、医生网页端和用户微信小程序端三大端，后端基于 SpringBoot+MyBatis 实现业务逻辑和数据交互，MySQL 作为数据存储介质，Vue 负责管理员和医生网页端前端界面开发，微信小程序原生框架实现用户端功能。
系统管理员网页端：面向医院管理人员，核心实现医院科室信息管理、医生账号管理、用户信息管控、系统日志查看、系统参数配置等功能，为系统运行提供基础数据支撑和全局管控。
医生网页端：面向医院医生，核心实现个人信息管理、排班设置、号源发布、预约信息查看、诊间叫号功能，支撑号源管理和就诊流程把控。
用户小程序端：面向患者，核心实现登录注册、修改密码、科室 / 医生筛选、预约挂号（无审核，号源有空 + 支付成功即生效）、预约 / 就诊记录查看功能，搭配微信消息通知（预约成功、就诊前 2 小时提醒），提升用户预约体验。
后端提供统一的接口服务、身份验证、数据校验、消息推送和日志管理等基础功能，保障三大端数据同步和系统稳定运行。
2.3 用户特征
系统管理员：使用电脑网页端，具备一定的计算机操作和系统管理能力，核心需求是统一维护医院科室基础信息、管理医生和用户账号、监控系统运行状态、查看系统日志，希望系统管理功能全面、操作便捷，能实现对系统的全局把控。
医生用户：使用电脑网页端，具备基本电脑操作能力，核心需求是灵活设置个人排班、自主发布号源、实时查看患者预约信息、便捷进行诊间叫号，希望系统后台操作简洁高效，能提升日常诊疗工作效率。
普通用户（患者）：使用微信小程序端，计算机 / 手机操作能力中等，核心需求是快速完成注册登录、精准筛选科室 / 医生、便捷预约挂号，能随时查看自身预约和就诊记录，希望操作流程简单、消息提醒及时，无需复杂学习成本。

分辨率：1366×768 及以上
网络环境：稳定的有线 / 无线网络，带宽≥2M
2.4.3 用户小程序端运行环境
微信版本：8.0 及以上
手机系统：Android 9.0+、iOS 13.0+

2.5 设计与实现约束
技术约束：必须基于 Java SpringBoot + MyBatis + MySQL + Vue 技术栈开发，遵循各框架开发规范和最佳实践。
功能约束：严格按照管理员、医生和用户的核心需求开发，预约流程无审核环节，满足 “号源有空 + 支付成功 = 预约生效” 的核心规则，消息通知仅包含预约成功和就诊前 2 小时提醒。
界面约束：管理员 / 医生网页端界面简洁专业，符合后台管理系统操作习惯，管理员端与医生端界面区分清晰；用户小程序端轻量化、操作便捷，遵循微信小程序设计规范，适配不同手机屏幕。
数据约束：所有数据持久化存储在 MySQL 数据库，保证数据一致性、完整性，对用户手机号、身份证号等隐私数据进行加密处理；管理员拥有数据最高操作权限，医生仅可操作个人相关数据，用户仅可查看个人数据。
开发约束：作为课程设计项目，需独立完成系统分析、设计、实现与测试，禁止直接使用现成开源系统修改，可参考资料但需自主设计核心代码和业务逻辑。
三、具体需求
3.1 功能需求
本系统功能需求按系统管理员网页端、医生网页端、用户小程序端、后端公共功能模块划分，各模块高内聚、低耦合，通过统一 API 接口实现数据交互，核心遵循 “预约无审核、支付即生效、消息及时推、管理统一化” 的业务规则。
3.1.1 后端公共功能模块
3.1.1.1 权限管理模块
角色划分：明确系统管理员、医生用户、** 普通用户（患者）** 三大角色，分配不同操作权限，无跨角色操作权限，其中管理员为最高权限角色。
Token 身份验证：管理员 / 医生 / 用户登录后生成唯一 Token，后续所有接口请求需携带 Token，Token 过期（2 小时）后需重新登录，未携带 / 无效 Token 直接拒绝请求。
密码加密：所有角色的密码均采用 BCrypt 加密算法存储，无明文存储，防止密码泄露。
接口权限控制：为每个 API 接口配置角色访问权限，如科室信息管理接口仅管理员可访问，号源发布接口仅医生角色可访问，预约挂号接口仅普通用户可访问，拒绝无权限角色请求并返回明确错误信息。
3.1.1.2 数据管理模块
通用数据操作：提供统一的数据库增删改查接口，支持医院、科室、医生、用户、排班、号源、预约等核心数据的基础操作，管理员拥有所有数据的增删改查权限，其他角色仅拥有对应数据的部分操作权限。
严格数据校验：对前端传入的所有参数进行合法性校验，如手机号格式、身份证号格式、预约时间与号源时段匹配性、支付金额与挂号费用一致性等，校验不通过返回明确错误提示。
数据分页与排序：对查询类接口实现分页功能（支持页码、每页条数参数），对科室、医生、用户、预约记录等查询结果支持按指定字段（如预约时间、医生姓名、用户注册时间）升序 / 降序排序，提升查询体验。
数据一致性保障：针对号源预约和支付操作，采用数据库事务处理，确保 “号源剩余数扣减” 和 “预约记录生成”、“支付状态更新” 原子性，避免号源超卖、预约成功未扣费等问题；管理员修改基础信息后，相关联数据实时同步更新。
3.1.1.3 消息推送模块
预约成功推送：用户完成支付且预约生效后，立即通过微信小程序模板消息向用户推送预约成功通知，包含医生姓名、科室、就诊时间、取号地点等核心信息。
就诊前提醒推送：系统自动在用户就诊时间前 2 小时，向用户推送微信模板消息提醒，包含就诊时间、医生、科室及就诊须知，避免用户错过就诊。
消息推送保障：推送失败时进行 1 次重试，记录推送日志，包含推送时间、用户 ID、推送内容、是否成功等信息，便于管理员和开发人员排查问题。
3.1.1.4 日志管理模块
操作日志：记录管理员、医生、用户的所有关键操作，如登录、注册、修改密码、预约挂号、支付、排班设置、号源发布、科室信息修改等，包含操作人 ID、操作人角色、操作内容、操作 IP、操作时间，管理员可按条件全局查询，医生 / 用户仅可查看个人操作日志。
错误日志：记录系统运行中的所有异常信息，如接口调用失败、数据库操作异常、参数校验错误、消息推送失败等，包含异常时间、异常类型、异常信息、请求接口、请求参数，仅系统管理员可查看和导出，便于系统维护和问题排查。
3.1.2 系统管理员网页端功能模块
系统管理员网页端基于 Vue 开发，采用侧边栏导航 + 主内容区 + 数据统计看板的后台管理布局，核心实现管理员登录、科室信息管理、医生账号管理、用户信息管理、系统日志查看、系统参数配置六大功能，拥有系统最高操作权限，实现对系统的全局管控，具体功能如下：
3.1.2.1 管理员登录模块
管理员登录：支持通过管理员账号 + 密码登录系统，账号为系统初始配置的超级管理员账号，唯一且不可修改，密码可由管理员自行修改。
登录安全：连续 5 次登录失败后，锁定账号 15 分钟，防止暴力破解；登录成功后生成 Token，用于后续接口身份验证；支持安全退出，清除登录状态和 Token。
密码修改：登录后可通过原密码 + 新密码修改登录密码，新密码要求长度≥8 位，包含数字 / 字母 / 特殊字符，提升账号安全性。
3.1.2.2 科室信息管理模块
科室列表查看：展示医院所有科室的完整信息，包含科室 ID、科室名称、科室简介、创建时间、更新时间，支持按科室名称模糊搜索、按创建时间排序，支持分页展示。
科室添加：管理员可手动添加医院科室，需填写科室名称、科室简介，提交后生成科室信息，同步至医生端和用户端，用户可在小程序端选择该科室进行医生筛选。
科室编辑 / 删除：支持对已添加的科室信息进行编辑（修改科室名称、简介），或直接删除未关联医生的科室；若科室已关联医生，不可直接删除，需先解除关联，防止数据不一致。
科室状态管控：支持对科室设置启用 / 禁用状态，禁用后的科室在医生端和用户端不可见，无法进行相关操作，启用后恢复正常。
3.1.2.3 医生账号管理模块
医生列表查看：展示全院所有医生的账号及基础信息，包含医生 ID、工号、姓名、性别、职称、所属科室、联系电话、账号状态、创建时间，支持按姓名 / 工号 / 科室搜索、按创建时间排序，支持分页展示，医生联系电话等隐私数据脱敏处理。
医生账号添加：管理员为新入职医生创建系统账号，需填写工号（唯一标识）、姓名、性别、职称、所属科室、联系电话，系统自动生成初始密码，账号创建后状态为启用，医生可使用工号 + 初始密码登录并修改密码。
医生账号编辑 / 删除：支持编辑医生的基础信息（姓名、职称、所属科室、联系电话），支持删除未发布任何号源的医生账号；若医生已发布号源 / 有预约记录，不可直接删除，仅可禁用账号。
医生账号状态管控：支持对医生账号设置启用 / 禁用状态，禁用后的账号无法登录系统，其已发布的未约满号源自动下架，已预约的待就诊记录保留；启用后账号恢复登录，医生可重新发布号源。
密码重置：支持为忘记密码的医生重置登录密码，重置后生成新的初始密码，管理员可将初始密码告知医生，医生登录后需立即修改。
3.1.2.4 用户信息管理模块
用户列表查看：展示所有注册用户的基础信息，包含用户 ID、手机号、真实姓名、身份证号（脱敏）、注册时间、最后登录时间，支持按手机号 / 姓名搜索、按注册 / 最后登录时间排序，支持分页展示，所有隐私数据均做脱敏处理。
用户信息查看：点击单条用户记录，可查看用户的详细注册信息和预约记录统计（累计预约次数、已就诊次数），不可修改用户的实名信息（姓名、身份证号）。
用户账号管控：支持对违规用户账号设置禁用 / 解封状态，禁用后的用户账号无法登录小程序，其未就诊的预约记录可正常就诊；解封后账号恢复登录功能。
预约记录全局查看：支持按科室、医生、就诊日期、预约状态查询全院所有用户的预约记录，用于医院预约数据统计和异常预约排查，仅可查看不可修改 / 删除。
3.1.2.5 系统日志查看模块
操作日志查询：展示全院所有角色的操作日志，支持按操作人 ID / 姓名、操作人角色、操作内容、操作时间范围筛选查询，支持按操作时间降序排序，支持日志导出为 Excel 文件，便于线下存档和统计。
错误日志查询：展示系统运行过程中的所有错误日志，支持按异常类型、异常时间范围、请求接口筛选查询，支持按异常时间降序排序，支持日志导出；点击单条错误日志，可查看详细的异常信息和请求参数，便于管理员排查系统故障。
日志清理：支持对超过 3 个月的操作日志和错误日志进行批量清理，释放数据库存储空间，清理前需进行二次确认，防止误操作。
3.1.2.6 系统参数配置模块
基础参数配置：支持配置系统基础参数，如微信消息推送开关、就诊提醒提前时间（默认 2 小时）、用户 / 医生连续登录失败锁定时间（默认 15 分钟）、Token 过期时间（默认 2 小时），修改后立即生效。
支付参数配置：支持配置微信支付相关基础参数（如商户号、支付密钥），参数加密存储，修改后需重启服务生效，确保支付功能正常运行。
数据备份配置：支持配置数据库自动备份时间（默认每日凌晨 2 点）、备份文件保留天数（默认 7 天），可手动触发数据库立即备份，备份文件存储在服务器指定目录，管理员可下载备份文件。
系统状态监控：展示系统实时运行状态，包含服务器 CPU、内存使用率、数据库连接数、当前在线人数（管理员 + 医生 + 用户）、今日预约总数、今日支付总数，为管理员系统维护提供数据支撑。
3.1.3 医生网页端功能模块
医生网页端基于 Vue 开发，采用侧边栏导航 + 主内容区的经典后台管理布局，核心实现医生登录、个人信息管理、排班设置、号源发布、预约信息查看、诊间叫号六大功能，仅可操作个人相关数据，数据与用户小程序端、管理员网页端实时同步，具体功能如下：
3.1.3.1 医生登录模块
医生登录：支持通过医生工号 + 密码登录系统，工号为管理员创建的唯一标识，密码为初始密码或医生自行修改后的密码。
登录安全：连续 5 次登录失败后，锁定账号 15 分钟，防止暴力破解；登录成功后生成 Token，用于后续接口身份验证；支持安全退出，清除登录状态和 Token。
密码找回：支持通过绑定手机号 + 短信验证码找回密码，手机号为管理员创建账号时填写的号码，确保密码找回的安全性。
3.1.3.2 个人信息管理模块
个人信息查看：登录后展示医生的基础信息，包含工号、姓名、性别、职称、所属科室、联系电话、个人简介、擅长领域，所有信息由管理员初始化配置，工号、所属科室不可修改。
个人信息修改：支持修改联系电话、个人简介、擅长领域，修改后提交即时生效，同步至数据库，管理员端和用户小程序端的医生详情页信息也实时更新。
密码修改：支持通过原密码 + 新密码修改登录密码，新密码要求长度≥6 位，包含数字 / 字母，修改成功后需重新登录。
3.1.3.3 排班设置模块
排班添加：医生可按具体日期 + 就诊时段设置个人出诊排班，支持选择单日多时段（如上午 8:00-12:00、下午 14:00-18:00），时段可自定义，添加后排班状态为 “已设置”，未发布号源前用户小程序端不可见。
排班编辑 / 删除：对未发布号源的排班，支持修改就诊日期、时段，或直接删除；对已发布号源的排班，不可直接修改 / 删除，需先下架对应号源后方可操作，避免号源与排班信息不一致。
排班查询：支持按日期范围（如本周、本月、自定义日期）查询个人所有排班记录，展示排班日期、时段、状态（已设置、已发布号源、已结束），支持按状态筛选，管理员端可同步查看。
临时停诊：对已发布号源的排班，支持设置临时停诊，停诊后剩余未预约号源立即下架，用户小程序端不可见，已预约的待就诊记录保留，系统不自动取消，由医生自行与患者沟通，停诊状态同步至管理员端。
3.1.3.4 号源发布模块
号源发布：基于已设置的排班（状态为 “已设置”），医生可发布对应时段的号源，需设置号源总数和挂号费用，发布后号源状态为 “已发布”，剩余号源数等于号源总数，同步至用户小程序端供用户预约，管理员端可查看号源发布信息。
号源下架：对已发布但未约满的号源，支持手动下架，下架后用户小程序端不可见，剩余号源数保留，可重新上架；号源约满后自动标注为 “已约满”，不可下架。
号源查询：支持按日期、时段、号源状态（已发布、已下架、已约满、已结束）查询个人发布的所有号源信息，展示就诊日期、时段、号源总数、剩余号源数、挂号费用、状态，数据实时同步（用户预约后剩余号源数即时扣减）。
号源编辑：对已发布未约满的号源，仅支持修改挂号费用，号源总数不可修改，避免影响已预约用户，修改后同步至用户小程序端和管理员端。
3.1.3.5 预约信息查看模块
预约记录实时查看：展示所有用户针对本人号源的预约记录，数据与用户小程序端、管理员端实时同步，包含预约编号、患者姓名、患者手机号（脱敏处理）、就诊科室、就诊时段、挂号费用、支付状态、预约状态（待就诊 / 已就诊）。
预约记录筛选：支持按就诊日期、预约状态、支付状态筛选查询，支持按就诊时间升序 / 降序排序，便于医生按时段查看预约患者。
预约详情查看：点击单条预约记录，查看详情信息，包含患者实名信息（姓名、身份证号脱敏）、预约时间、支付时间、预约编号、取号地点，无修改 / 删除预约记录的权限。
就诊状态更新：患者到院就诊后，医生可点击 “标记为已就诊”，将预约状态从 “待就诊” 更新为 “已就诊”，更新后同步至用户小程序端的历史记录和管理员端的全局预约记录。
3.1.3.6 诊间叫号模块
待叫号列表：按就诊时段展示当前时段的待就诊预约记录，按用户预约的先后顺序排序，包含患者姓名、预约编号、就诊序号，实时同步预约状态（待就诊 / 已就诊）。
诊间叫号：医生点击患者姓名旁的 “叫号” 按钮，系统展示文字叫号提示（如 “请 XXX 患者到 XX 诊室就诊”），支持连续叫号，叫号记录自动保存（包含叫号时间、患者姓名、就诊序号），管理员端可查看叫号记录。
过号处理：对未及时就诊的患者，医生可点击 “过号” 标记，标记后患者记录可移至当前时段待叫号列表末尾，支持重新叫号。
就诊完成确认：患者就诊结束后，医生点击 “就诊完成”，将该患者预约状态更新为 “已就诊”，同步至预约信息查看模块、用户小程序端和管理员端。
3.1.4 用户小程序端功能模块
用户小程序端基于微信小程序原生框架开发，遵循 “极简操作” 原则，核心实现登录注册、修改密码、预约挂号、记录查看四大核心功能，预约流程无审核，号源有空且支付成功即预约生效，配套微信消息通知，具体功能如下：
3.1.4.1 登录注册模块
用户注册：支持通过手机号 + 短信验证码 + 密码完成注册，需校验手机号唯一性（未注册方可注册），密码要求长度≥6 位，包含数字 / 字母，注册成功后自动登录，注册信息同步至管理员端。
用户登录：支持两种登录方式，一是手机号 + 密码登录，二是手机号 + 短信验证码快捷登录，登录失败给出明确提示（如手机号未注册、密码错误、验证码无效 / 过期）。
信息完善：首次登录后引导用户完善实名信息（真实姓名、身份证号），实名信息仅用于挂号身份验证，加密存储，不可随意修改，同步至管理员端。
3.1.4.2 个人信息与密码修改模块
个人信息查看：在个人中心展示用户的实名信息（真实姓名、身份证号，身份证号做脱敏处理，如 410XXX1990XXXX1234）、手机号，不可修改实名信息和手机号，管理员端可查看完整脱敏信息。
密码修改：支持通过原密码 + 新密码修改登录密码，或通过手机号 + 短信验证码找回 / 修改密码，新密码需满足注册时的复杂度要求，修改成功后需重新登录。
3.1.4.3 预约挂号模块
科室 / 医生筛选：展示管理员端配置的所有启用科室列表，用户选择对应科室后，展示该科室下的所有启用医生列表，医生列表包含医生姓名、职称、擅长领域、出诊时间，支持医生姓名模糊搜索。
号源查看：点击医生进入详情页，展示该医生的近期排班及对应时段的号源信息，包含就诊日期、就诊时段、剩余号源数、挂号费用，号源约满后标注 “已约满”，不可预约，号源信息与医生端、管理员端实时同步。
预约操作：用户选择有剩余号源的时段后，确认就诊人信息（仅本人，基于实名信息），核对预约信息（医生、科室、就诊时间、挂号费用），确认后进入支付环节。
在线支付：对接微信支付，支持用户通过微信完成挂号费用支付，支付页面展示支付金额、收款主体，支付成功后立即生成预约记录，预约状态为待就诊，同时扣减对应号源剩余数，支付信息同步至医生端和管理员端；支付失败则预约作废，号源数不扣减。
预约生效规则：无任何审核环节，满足号源有剩余 + 支付成功两个条件，预约立即生效，系统自动推送预约成功微信消息，预约记录同步至医生端和管理员端。
3.1.4.4 就诊 / 预约记录查看模块
记录分类展示：在个人中心分待就诊记录和历史记录两大板块展示，历史记录包含已就诊、已取消的预约记录，所有记录与医生端、管理员端实时同步。
待就诊记录：展示所有预约生效且未到就诊时间的记录，包含医生姓名、科室、就诊时间、挂号费用、预约状态（待就诊），点击可查看预约详情（取号地点、就诊须知、预约编号）。
历史记录：展示所有已就诊、已取消的记录，包含医生、科室、就诊时间、挂号费用、最终状态（已就诊 / 已取消），支持按时间范围（如近 1 个月、近 3 个月）筛选查询。
记录详情：所有记录点击均可查看详情，包含预约编号、就诊人信息、预约时间、支付时间、挂号费用、取号地点等，无修改 / 删除权限，仅做查看使用，管理员端可全局查看所有用户记录。
3.1.4.5 个人中心模块
核心功能入口：包含待就诊记录、历史记录、修改密码、我的实名信息等快捷入口，布局简洁，操作便捷。
系统设置：包含小程序缓存清理、消息推送开关（仅控制是否接收微信消息，默认开启）、关于系统等基础功能。
退出登录：支持安全退出登录，清除小程序端登录状态和 Token，退出后需重新登录方可操作。
3.2 非功能需求
3.2.1 性能需求
响应时间：后端接口平均响应时间≤1s，最大响应时间≤3s；管理员 / 医生网页端页面加载时间≤3s，用户小程序端页面加载时间≤2s；号源扣减、预约记录生成、支付状态更新等核心操作响应时间≤0.5s；管理员批量查询 / 导出数据时，响应时间≤5s。
并发处理：系统支持至少 50 个并发用户、10 名在线医生、2 名在线管理员同时操作，并发情况下接口响应时间无明显增加，数据无丢失、错乱，无号源超卖问题。
数据处理能力：单医生单时段最大支持发布 200 个号源，系统可快速处理号源预约、支付、剩余数扣减等操作，无延迟；管理员可快速处理全院上千条用户 / 医生数据和上万条日志数据。
稳定性：系统 7×24 小时稳定运行，非维护时间内故障停机时间≤1 小时 / 月，故障恢复时间≤30 分钟，核心业务（预约、支付、叫号、基础信息管理）无单点故障。
3.2.2 可靠性需求
数据一致性：核心操作（预约 + 支付、号源发布 + 排班、科室修改 + 医生关联）采用数据库事务处理，确保原子性，要么全部成功，要么全部回滚，避免数据不一致；多用户并发预约同一号源时，通过数据库行锁保障剩余号源数准确扣减，无超卖；管理员修改数据后，各端数据实时同步。
数据容错性：对管理员 / 医生 / 用户的非法操作、异常输入（如输入非数字的挂号费用、选择已过期的预约时间、添加重复科室名称）进行容错处理，给出明确错误提示，不会导致系统崩溃、页面卡死或数据损坏。
数据备份与恢复：系统支持 MySQL 数据库每日自动备份，管理员可手动触发即时备份，备份文件存储在独立目录，保留近 7 天备份；当数据丢失、损坏时，管理员可通过备份文件快速恢复，数据恢复率≥99%，恢复时间≤1 小时。
3.2.3 安全性需求
隐私数据保护：用户的身份证号、手机号，医生的联系电话等隐私数据，采用 AES 加密算法存储在数据库，传输过程采用 HTTPS 协议，展示时做脱敏处理（如手机号 138****1234）；仅管理员可查看全院脱敏后的隐私数据，医生仅可查看个人隐私数据，用户仅可查看自身隐私数据。
接口安全：所有后端接口添加请求频率限制（单 IP 每分钟≤60 次），防止恶意刷接口；接口请求需携带有效 Token，未携带、过期、无效 Token 均拒绝请求；对支付、密码修改、账号禁用等敏感接口，添加请求参数签名验证，防止参数篡改。
支付安全：对接微信支付官方接口，遵循微信支付安全规范，支付参数加密传输，支付结果通过微信支付异步通知校验，避免伪造支付成功信息；支付记录与预约记录一一绑定，无支付记录则无法生成有效预约，管理员可查看全院支付记录。
账号安全：所有角色的密码采用 BCrypt 加密，无明文存储；连续多次登录失败后账号锁定，防止暴力破解；Token 采用随机字符串生成，包含时间戳，过期后自动失效，防止 Token 盗用；管理员账号为最高权限，仅可由专人操作，密码复杂度要求更高。
操作安全：管理员执行删除、禁用、密码重置、日志清理等高危操作时，需进行二次确认，防止误操作；所有高危操作均记录详细操作日志，便于追溯。
3.2.4 易用性需求
管理员 / 医生网页端：界面布局清晰，侧边栏导航按功能模块分类，常用功能设置主内容区快捷入口；所有操作均有明确提示（如保存成功、删除确认、高危操作二次提醒）；管理员端提供数据统计看板，直观展示系统运行状态；无需专业培训，管理员 / 医生 5 分钟内可上手核心操作。
用户小程序端：操作流程极简，从 “选择科室 / 医生” 到 “完成支付预约” 的步骤≤4 步；界面元素简洁，按钮、文字清晰醒目，适配不同尺寸手机屏幕；首次使用时展示简易操作引导，核心功能（预约、查记录）入口置于首页，一目了然。
3.2.5 可维护性需求
代码规范：前后端代码遵循统一编码规范，Java 代码遵循阿里巴巴 Java 开发手册，Vue 代码遵循 Vue 官方编码规范，小程序代码遵循微信小程序开发规范；所有类、方法、核心业务逻辑添加详细注释，注释率≥80%，便于阅读、修改和扩展。
模块化设计：系统采用模块化设计，各功能模块独立开发、独立部署，模块之间通过标准化 API 接口交互，如管理员的科室管理模块仅依赖数据管理模块，修改某一模块不会影响其他模块正常运行。
日志完善：操作日志和错误日志记录详细、规范，管理员可按多条件全局查询和导出，便于开发和维护人员快速定位问题、排查故障；医生 / 用户可查看个人操作日志，便于追溯自身操作。
3.2.6 可扩展性需求
功能扩展：系统预留扩展接口，便于后续添加新功能，如用户端的取消预约、医生端的预约统计、管理员端的数据分析报表、多医院对接等，无需对现有核心代码进行大规模修改。
数据扩展：数据库表结构设计预留扩展字段，如用户信息表、医生信息表、科室信息表添加 “扩展字段 1/2”，便于后续新增数据项；支持数据库分表，当预约记录、日志数据量增大时，管理员可按时间分表，提升查询性能。
端扩展：后端接口采用 RESTful API 设计，便于后续扩展 H5 端、APP 端，仅需开发对应前端界面，无需修改后端接口逻辑；管理员可对新增端进行权限配置。
3.3 数据需求
3.3.1 核心数据实体及属性
本系统核心数据实体围绕管理员、用户、医生、科室、排班、号源、预约、支付设计，各实体属性如下，所有实体均设置主键（自增 ID），外键关联相关实体，保证数据关联性；管理员拥有所有实体数据的最高操作权限：
管理员信息表（admin）：管理员 ID（主键）、管理员账号、密码（BCrypt 加密）、姓名、联系电话、创建时间、最后登录时间。
用户信息表（user）：用户 ID（主键）、手机号、密码（BCrypt 加密）、真实姓名、身份证号（AES 加密）、创建时间、更新时间、最后登录时间、账号状态（0 - 启用，1 - 禁用）。
科室信息表（dept）：科室 ID（主键）、科室名称、科室简介、创建时间、更新时间、科室状态（0 - 启用，1 - 禁用）。
医生信息表（doctor）：医生 ID（主键）、工号、姓名、性别、职称、科室 ID（外键，关联科室表）、联系电话（AES 加密）、个人简介、擅长领域、密码（BCrypt 加密）、创建时间、更新时间、账号状态（0 - 启用，1 - 禁用）。
排班信息表（schedule）：排班 ID（主键）、医生 ID（外键，关联医生表）、就诊日期、就诊时段、排班状态（0 - 已设置，1 - 临时停诊）、创建时间、更新时间。
号源信息表（number_source）：号源 ID（主键）、排班 ID（外键，关联排班表）、号源总数、剩余号源数、挂号费用、号源状态（0 - 已发布，1 - 已下架，2 - 已约满，3 - 已结束）、创建时间、更新时间。
预约信息表（appointment）：预约 ID（主键）、号源 ID（外键，关联号源表）、用户 ID（外键，关联用户表）、患者姓名、患者身份证号（AES 加密）、挂号费用、预约状态（0 - 待就诊，1 - 已就诊）、预约时间、支付时间、创建时间、更新时间。
支付信息表（pay）：支付 ID（主键）、预约 ID（外键，关联预约表）、支付金额、支付方式（0 - 微信支付）、支付状态（0 - 未支付，1 - 已支付）、微信交易单号、创建时间、更新时间。
消息推送日志表（msg_log）：日志 ID（主键）、用户 ID（外键，关联用户表）、推送类型（0 - 预约成功，1 - 就诊前提醒）、推送内容、推送状态（0 - 失败，1 - 成功）、推送时间、重试次数。
操作日志表（operate_log）：日志 ID（主键）、操作人 ID、操作人角色（0 - 用户，1 - 医生，2 - 管理员）、操作内容、操作 IP、操作时间。
错误日志表（error_log）：日志 ID（主键）、异常类型、异常信息、请求接口、请求参数、操作 IP、异常时间。
系统参数表（sys_param）：参数 ID（主键）、参数名称、参数值、参数说明、修改时间、修改人 ID。
3.3.2 数据关系
一对多关系：科室与医生（一个科室对应多个医生）、医生与排班（一个医生对应多个排班）、排班与号源（一个排班对应多个号源）、号源与预约（一个号源对应多个预约）、预约与支付（一个预约对应一个支付）、用户与预约（一个用户对应多个预约）、管理员与科室 / 医生 / 用户（一个管理员管理所有科室 / 医生 / 用户）。
多对一关系：医生与科室、排班与医生、号源与排班、预约与号源、预约与用户、支付与预约。
数据关联约束：外键关联采用 “级联查询、不级联删除”，如管理员删除科室时，不自动删除该科室下的医生，防止数据误删；医生删除排班时，需先删除该排班下的号源，防止号源与排班数据不一致；管理员禁用医生账号时，其已发布号源自动下架，不删除预约记录。
3.3.3 数据存储与管理
存储引擎：所有数据库表采用InnoDB存储引擎，支持事务和行锁，保障核心操作的原子性和并发数据的一致性。
索引设计：对常用查询字段建立索引，如用户表的手机号、医生表的工号 / 科室 ID、科室表的科室名称、排班表的医生 ID + 就诊日期、号源表的排班 ID + 号源状态、预约表的用户 ID / 医生 ID + 预约状态，提升查询效率；管理员端高频查询的日志表按操作人角色 / 操作时间建立索引。
隐私数据处理：用户身份证号、手机号，医生联系电话等隐私数据，入库前采用 AES 加密，出库后解密展示（脱敏处理），加密密钥独立存储，不写入代码；仅管理员可查看全院脱敏隐私数据。
数据清理：系统日志表（操作日志、错误日志、消息推送日志）保留近 3 个月数据，超过 3 个月的由管理员手动清理或自动归档；管理员可对冗余的预约 / 支付记录进行归档处理，避免单表数据量过大影响查询性能。
数据权限控制：数据库层面通过角色权限限制数据访问，管理员可访问所有表数据，医生仅可访问与本人 ID 关联的表数据，用户仅可访问与本人 ID 关联的表数据。
3.3.4 数据交互
数据格式：前后端之间通过JSON格式进行数据交互，接口返回数据包含统一格式：状态码、提示信息、数据体，无数据时数据体为 null，具体格式如下：
json
{
  "code": 200, // 200-操作成功，400-参数错误，401-未登录/Token过期，403-无权限，500-服务器内部错误
  "msg": "操作成功", // 错误时返回具体错误信息
  "data": {} // 业务数据体，按需返回
}
传输协议：所有前后端接口、小程序与后端接口均采用HTTPS协议传输，防止数据在传输过程中被窃取、篡改。
第三方数据交互：与微信支付、微信小程序模板消息的对接，遵循微信官方 API 接口规范，采用 XML/JSON 格式交互，通过微信提供的签名机制进行身份验证和参数校验，相关参数由管理员在系统参数配置模块统一维护。
跨端数据同步：管理员修改科室 / 医生信息后，实时同步至医生端和用户端；医生发布 / 下架号源、更新预约状态后，实时同步至用户端和管理员端；用户完成预约 / 支付后，实时同步至医生端和管理员端，确保各端数据一致性。
四、接口需求
4.1 接口设计原则
所有接口采用RESTful API设计风格，使用 GET（查询）、POST（新增）、PUT（修改）、DELETE（删除）请求方法，语义清晰。
接口地址统一前缀：系统管理员端接口前缀为/api/admin，医生端接口前缀为/api/doctor，用户小程序端接口前缀为/api/user，后端公共接口前缀为/api/common。
接口参数：查询类接口参数通过 URL 传参，新增 / 修改 / 删除类接口参数通过请求体传参（JSON 格式），所有接口均需携带 Token 请求头（token: xxx），管理员端高危操作接口需额外携带操作人签名。
接口返回：所有接口返回统一的 JSON 格式，包含 code、msg、data，状态码遵循 HTTP 规范，业务状态码在返回体中定义；管理员端批量查询接口返回数据包含总条数、分页数据，支持导出。

4.3 第三方对接接口
微信小程序登录接口：https://api.weixin.qq.com/sns/jscode2session，通过小程序端的 code 换取微信 session_key，用于短信验证码校验和微信授权，相关配置由管理员维护。
微信支付统一下单接口：https://api.mch.weixin.qq.com/pay/unifiedorder，提交预约订单信息，获取微信支付参数，供用户小程序端调起微信支付，商户参数由管理员在系统参数配置模块统一配置。
微信支付异步通知接口：自定义接口/api/user/pay/wx/notify，微信支付成功后异步回调该接口，更新支付状态和预约状态，同步至医生端和管理员端。
微信小程序模板消息推送接口：https://api.weixin.qq.com/cgi-bin/message/wxopen/template/send，提交用户 openid、模板 ID、推送内容，向用户推送微信消息，推送开关由管理员配置。
